Building a testing framework for Plants By WebSphere

Creating a testing framework for a monolithic application running on WAS classic in order to get fast feedback, and confidence, in evolving the application to microservices that will run on Liberty.

In a previous article, Testing microservices, we laid out some rules of engagement for testing applications in the context of evolving a monolithic application. This came about from our experiences of testing the Plants by WebSphere application, where we were running it in WAS classic before starting to move parts of it into microservices that were running on Liberty.

This article explores in more depth how we developed these thoughts, and goes through our creation of a framework to allow us to test the monolithic application running on WAS classic to get fast feedback and confidence for the evolution to microservices.

Getting started

When we started we were doing manual testing. This was mainly so that we could understand what the application was doing. We examined existing documentation and studied the source code. We also played with the existing Maven build system. This was a useful period for us because we began to develop a strong understanding of what the application was doing and where we could start to define areas that might need to be evolved into their own microservice.

Moving to automation

Once we had a reasonable idea of the shape of the application we began looking at separating the Image service. While we were happy with the simplicity of the changes, we realised that this wasn’t going to hold true in future. And now we were looking at Bluemix as our future deployment platform and we were starting to think about how we could run tests. So we looked at building some simple automation around our monolithic application.

The automation process we wanted would do the following:

  1. Build the application using the existing build system.
  2. Run the unit tests.
  3. Deploy the application to a test server.
  4. Run the integration (functional) tests on the test server.
  5. If the tests pass, pass the build; otherwise fail the build.

We decided to focus on integration testing because, fundamentally, we want to make sure that the outside behaviour remains the same. We are more concerned with parity of the externals because they represent the impact on the customer if you break something. Ensuring parity of the internals probably doesn’t make sense, because you know the internals will have to change in order for you to move to microservices. It’s always a good idea, though, as part of new modules and updating old modules, to add unit tests.

Once we started looking at how to build a framework to support this we hit a few issues that would negatively affect the microservices developer experience and productivity:

  • Installing WAS classic requires more steps than installing Liberty.
  • Due to the larger amount of function loaded at start-up, starting a WAS classic server can take significantly longer than starting a WAS Liberty server.
  • There is no current framework to deploy applications to a WAS classic server from Maven.

This led us to a question: is it reasonable to not run the tests? If there are separate developers dedicated to maintaining the original WAS application, you don’t want to make their life harder. Also, the aim of building the automated test framework was to have a fast turnaround in developing and testing the microservice so that we can quickly be confident that we haven’t broken anything in the process of evolving the monolith application to microservices. If the server for the monolith application is slow to install and start, the whole framework becomes slower. So maybe some manual steps would be acceptable, as long as they’re infrequent and simple to understand.

Here are the manual steps we came up with:

  1. Install WAS classic.
  2. Set up a server.
  3. Start the server.
  4. Configure the monitoredDeployableApps directory.
  5. Run the build of the monolith application, specifying the location of the WAS server.

If the location of the WAS classic server installation isn’t specified, the tests don’t run. If it is specified, we deploy the app to the test server, check it has started, and then run the tests.

This was a great solution for a fairly simple app where we had no automated deployment before. However, it’s worth noting that if you already have a setup to deploy your monolithic application, a lot of this may be business-as-usual, and you don’t need to worry. If you find yourself in a situation with no automated deployment method and you can’t use the monitoredDeployableApps directory, another route is to automate the application deployment using the wsadmin scripting mechanism. The Knowledge Center contains more info on how to do this.

Deciding to test for parity

We started off using HTMLUnit and added a simple test to ensure the welcome page appeared. And almost immediately we hit an error:

com.gargoylesoftware.htmlunit.ScriptException: TypeError: Cannot find function deselectMenu in object [object]

We tracked this failure down to some boilerplate text in all the HTML files that we send back to the browser. The file containing the deselectMenu function is only available in the orderInfo page but it’s added to lots of other HTML pages. HTMLUnit is right and this code is wrong. But how do we fix it? Technically, the right thing to do is to fix the HTML. Either remove the reference (since it can’t actually be doing anything, right?) or add the function in; someone asked for it so they must have expected it to do something.

This led us to one of our key points: we shouldn’t make changes while we are still figuring out what the app is doing. The primary reason for the test is to ensure that a page is being returned. The original pages have JavaScript issues so, at this stage, it doesn’t make sense to check that the JavaScript parses OK. Hence we disabled the JavaScript with this line:

webClient.getOptions().setJavaScriptEnabled(false);

If you are anything like us, that’s going to feel uncomfortable. We now know something is broken but we’ve essentially hidden it from the testing. The compromise we accepted is that this shouldn’t be the long-term solution. Once our code has sufficient testing and we can feel confident altering things, we should go back and fix this. Until then, we need to focus on what’s changed, not on what’s right.

This of course still leaves that discomfort around this stuff we “should go back and fix”. One aspect we have considered is to add another test suite of tests that will fail but do what we want, effectively building out the behaviour we want to see as we go. Once you have migrated the app you can then go back and re-enable those tests. We’ll be looking to do this to our tests soon.

The next stages

So far we have the monolithic application tested. Separately, when we added the Image service, we added tests to that. However, we still have tests from the monolithic application that now need to change or move. That then becomes our focus and introduces new concerns we need to think about, such as:

  • How will we build the infrastructure to test the two separate parts?
  • What do we need to do to test how two services interact?
  • How do we assess the application’s behaviour when services go away? Or, perhaps more importantly, when they come back?

These are real issues but they are well documented in the microservices world and there are good strategies to handle them. The key things to be aware of are to understand how the new microservices will be tested and where the boundaries are.

We plan to follow this up with more content on testing microservices, so stay tuned for more. If you have any thoughts, please leave comments on this page.
