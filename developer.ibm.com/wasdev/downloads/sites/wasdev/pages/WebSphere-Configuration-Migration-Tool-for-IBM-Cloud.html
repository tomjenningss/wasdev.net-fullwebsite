WebSphere Configuration Migration Tool for IBM Cloud

Supports migrating to IBM Cloud (public), to IBM Cloud Private, and now to Docker.

The tool allows you to select from one of two wizards. The "Virtual Machine" wizard allows you to migrate your on-premises WebSphere cell, server or application to virtual machines running in the IBM Cloud (public or private). The "Container" wizard allows you to migrate an application from a WebSphere cell into its own standalone server running in a Docker container by generating the necessary Docker build artifacts.

For public IBM Cloud, you will need an IBM Cloud account with access to the WebSphere Application Server service. For IBM Cloud Private, contact the administrator of your company's installation to set up proper access to the WebSphere Application Server VM Quickstarter service.

When starting the "Virtual Machine" wizard, you are given a high-level overview of the entire on-premises to IBM Cloud migration process, which consists of two main phases:

  1. Creating and assigning a WebSphere service instance to handle the uploaded configuration data from the selected on-premises WebSphere Cell.
  2. Provisioning the virtual machines, which includes the migration of the uploaded configuration data assigned to the service instance.

Phase 1 of the process is handled by the easy to use "Virtual Machine" wizard which guides you through the 6 steps. The last step of phase 1 provides a link to the provisioning page for the assigned service instance. Using this link, you will kick off phase 2 which provisions a virtual machine for each node in the cell. Provisioning also includes the migration of the node’s configuration data.

When starting the "Container" wizard you are also given a high-level overview of the migration process, which consists of two main phases:

  1. Selecting an application from your WebSphere cell and generate the Docker build artifacts.
  2. Modify the artifacts, build the Docker image and publish it.

Docker migrations only support migrating into a standalone server. It does not support migrating a full Network Deployment cell. During the Docker build process the application and its associated configuration data is extracted from the old WebSphere cell and migrated into the standalone server.

The applicaiton selective migration feature used by the Docker migration process is also available for IBM Cloud migrations. The IBM Cloud migration process also allows you to selective migrate a single server and its applications from a Network Deployment cell into its own standalone server.

Only WebSphere v9 is available when you migrate to the IBM Cloud or to Docker. Both wizards allow you to migrate from WebSphere release v7, v8, v855 or even WebSphere v9. They utilizes the “cloning” migration process that was delivered as part of WebSphere v9. “Cloning” allows your on-premises WebSphere cell to continue running both during and after the cell has been migrated to the IBM Cloud. Once the migration completes, both the old and new cells are independent and can run concurrently.

Avoid the hassles of hardware and software management by cloning your WebSphere cell to the IBM Cloud. Use it to preview how your applications perform under WebSphere v9. Or to see how to tune your configuration for v9. Use it to practice and work out your future on-premises migration process. Provide a clone for a development team to test out their applications under v9. Clone once, clone many, use it for a short time, or for as long as you need. This tool makes it easier to achieve your WebSphere v9 goals.

Installation

  1. Download the WebSphere Configuration Migration Tool for IBM Cloud by clicking Download.
  2. Extract the file into a new directory.
  3. Be sure the PATH environment variable is set correctly.
    Example on Windows: set PATH=c:\java\bin;%PATH% Example on Mac/Linux: export PATH=/java8/bin:$PATH
  4. To start the wizard run this command from the directory containing CloudMigration.jar: java -jar CloudMigration.jar
