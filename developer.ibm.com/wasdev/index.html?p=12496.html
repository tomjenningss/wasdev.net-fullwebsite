The Liberty Repository is the place to find Liberty content such as runtimes and features. The Liberty Repository is hosted by IBM but you can, alternatively, install an on-premise version of the repository for greater control over the features available to your organisation. Here’s how…

Why do you need an on-premise Liberty repository?

The Liberty Repository is the place to find Liberty content such as runtimes and features. You can download a Liberty runtime from the repository and then, perhaps at a later date, install additional features and bundles using Liberty’s installUtility command. The Liberty Repository is hosted by IBM, so it is available from any host that has internet access.

The repository is powerful and may be all that you need. Some organizations, however, have additional requirements to control and govern the precise set of features that can be installed on their Liberty runtimes. Perhaps they author their own Liberty features and want to make them available within their organization. The Liberty Asset Repository Service, or LARS, is designed to meet these requirements.

LARS is an on-premise Liberty repository that runs as a Java EE web application on top of Liberty to which you can upload Liberty features. You can configure the repository clients (Installation Manager, WebSphere Developer Tools, or installUtility) to install features and bundles directly from an instance of LARS that you control.

This article shows you how to install, populate, and use a simple LARS repository. Our example applies to Linux or Mac but similar steps are applicable to the Windows platform.

Getting and installing LARS

Installing Liberty

LARS runs on top of Liberty so the first step is to install a Liberty runtime if you do not already have one. You just need the kernel package of the application server with no runtime features. Once you have installed the Liberty kernel package, we’ll use installUtility to install the features that LARS depends upon directly from the Liberty Repository, before then installing LARS itself.

  1. Download the Liberty Kernel.
  2. Install Liberty by unzipping it: 
    unzip -d ~/LARS wlp-kernel-19.0.0.9.zip

Installing MongoDB

LARS uses MongoDB as a data store. See the LARS prerequisites page for details of which versions are supported

  1. Download MongoDB.
    Consult the MongoDB manual for information on installing and configuring MongoDB. In this example, we use an un-secured MongoDB instance that only listens for connections from localhost and is not accessible from any other machine on the network. We achieve this by setting bind_ip = 127.0.0.1 in mongod.conf. For a production deployment, consult the MongoDB documentation to learn how to configure MongoDB security.
  2. Ensure that MongoDB is started and that you know what port it is using; you’ll need that information later.

Downloading and installing LARS

For more tail, see the LARS install instructions on github

  1. Download LARS from the Liberty Repository.
  2. Now install LARS by unzipping the zip file that you downloaded into the wlp/usr directory of your Liberty install: 
    ~/LARS/wlp/usr
unzip ~/Downloads/larsServer.zip
  3. Now install the pre-requisite features into your liberty runtime with the following commands: 
    cd ~/LARS/wlp/bin

./installUtility install larsServer

    You’ll see messages like these confirming that the features have been installed:

    Establishing a connection to the configured repositories...
This process might take several minutes to complete.

....
....

One or more features installed successfully: cdi-1.0 json-1.0 servlet-3.0 jaxrs-1.1 ssl-1.0 jndi-1.0 appSecurity-2.0 distributedMap-1.0.
Start product validation...
Product validation completed successfully.

LARS is now installed. You will see that a new server, larsServer, has been created in your Liberty runtime. The next step is to configure it.

Configuring the LARS server

If you look at ~/LARS/wlp/usr/servers/larsServer/server.xml, you can see that it has multiple sections commented out. For a production deployment of LARS, you can configure these to your own requirements. For now, replace the comments with a simple configuration in order to get your LARS server set up quickly.

Open ~/LARS/wlp/usr/servers/larsServer/server.xml in a text editor and make the following edits:

  1. Create a user registry or uncomment the example basicRegistry. For more details, see configuring a basicRegistry. 
        <basicRegistry id="defaultRegistry" realm="default">
        <user name="admin" password="yourPasswordHere"/>
        <group name="Administrators">
            <member name="admin"/>
        </group>
    </basicRegistry>

  2. Add host="*" to the httpEndpoint element if you want LARS to be used by Liberty runtimes on other hosts: 
        <httpEndpoint host="*" httpPort="9080" httpsPort="9443" id="defaultHttpEndpoint"/>

  3. Add security-role mappings to the webApplication element. For more information, see Configuring authorization for Liberty applications.

    Alternatively, uncomment the example role-mappings in server.xml, which grant read-access to everyone (including users who do not authenticate) but only allows users in the Administrators group to write to LARS:

        <webApplication id="com.ibm.ws.lars.rest" location="larsServer.war" name="com.ibm.ws.lars.rest" contextRoot="/">
        <classloader commonLibraryRef="mongo-lib" apiTypeVisibility="spec,ibm-api,api,third-party"/>
        <application-bnd>
            <security-role name="User">
                <special-subject type="EVERYONE"/>
            </security-role>
            <security-role name="Administrator">
                <group name="Administrators"/>
            </security-role>
        </application-bnd>
    </webApplication>

Starting LARS

From the ~/LARS/wlp/bin directory, start LARS:

    cd ~/LARS/wlp/bin/

    ./server start larsServer

You’ll see messages like these confirming that the server has started:

    Starting server larsServer.
    Server larsServer started with process ID 19564.

Verify that LARS is running by going to the following URL in your web browser: http://localhost:9080/ma/v1/assets

If you see [] (empty square brackets) then LARS is running (and does not contain any content yet):


Downloading larsClient

You need a way to load content into your LARS server. larsClient is a commandline utility that lets you upload, search for, and delete Liberty features or an Addon bundle, such as the javaee7Bundle. larsClient does not need to be run on the same host as the Liberty server (although that’s what you’ll do here).

  1. Download the LARS Client from the Liberty Repository.
  2. Unzip it on the same machine as your LARS server: 
    unzip -d ~/LARS/larsClient larsClient.zip

Downloading some content to store in LARS

Next, you need to download some content from the Liberty Repository to store in LARS.

  1. Download javaee7Bundle, a bundle of features that together implement Java EE 7. Use installUtility to download the bundle: 
    mkdir -p ~/LARS/javaee7Bundle

./installUtility download javaee7Bundle  --verbose --location=$HOME/LARS/javaee7Bundle

    You’ll see messages like these confirming that the bundle has been installed:

    Establishing a connection to the configured repositories...
This process might take several minutes to complete.

installUtility has downloaded the bundle and all of the features that it contains to the directory ~/LARS/javaee7Bundle. The bundle itself is ~/LARS/javaee7Bundle/addons/19.0.0.9/com.ibm.websphere.appserver.javaee7Bundle.esa and the features can be seen in the directory ~/LARS/javaee7Bundle/features/19.0.0.9.

Upload assets to LARS using larsClient

  1. Point larsClient to the directory that contains the features that you downloaded. This will upload all of the .esa files from that directory, populating LARS with all of the Java EE 7 features. Note that this operation will take several minutes to complete. 
    ~/LARS/larsClient/bin/larsClient upload ~/LARS/javaee7Bundle/features/19.0.0.9 --url=http://localhost:9080/ma/v1 --username=admin --password=yourPasswordHere

    You’ll see messages like these showing that each .esa has been uploaded to LARS:

    1 of 164: Uploading /home/liberty/LARS/javaee7Bundle/features/19.0.0.9/com.ibm.websphere.appserver.transactionContext-1.0.esa ... done
2 of 164: Uploading /home/liberty/LARS/javaee7Bundle/features/19.0.0.9/com.ibm.websphere.appserver.javax.ejb-3.2.esa ... done
3 of 164: Uploading /home/liberty/LARS/javaee7Bundle/features/19.0.0.9/com.ibm.websphere.appserver.websocketCDI-1.2.esa ... done

You now have a LARS repository that is populated with content. A Liberty runtime can install Java EE 7 features from that repository.

Configuring another Liberty server to use LARS

Create a new instance of Liberty, into which you can install the Java EE 7 features. In this example, you’ll do so on the same host that LARS is running on but, for production use, you would be much more likely to have Liberty servers running on many different hosts.

  1. Unzip the Liberty kernel package again to create a new instance of Liberty: 
    unzip -d ~/liberty-server wlp-kernel-19.0.0.9.zip
  2. Now configure your Liberty server to use LARS by creating a repositories.properties file in Liberty’s etc directory: 
    mkdir -p ~/liberty-server/wlp/etc
  3. Create repositories.properties using your favorite text editor (other editors are available): 
    vi ~/liberty-server/wlp/etc/repositories.properties

    If you need a sample repositories.properties file then you can run installUtility viewSettings, which will give you a sample that you can customize with your own values. For now, though, use the following contents:

            # Use of the online IBM WebSphere Liberty Repository
        # Set the useDefaultRepository property to false to prevent
        # Asset Manager from accessing the internet to connect to the
        # IBM WebSphere Liberty Repository repository.
        useDefaultRepository=false
        
        # Use of custom repositories
        # Provide a repository name and either the file path or URL to each
        # custom repository containing Liberty assets and service.
        # The repositories are accessed in the order that they are specified.
        
        # Specify the name and URL to hosted repositories
        LARS.url=http://localhost:9080/ma/v1
        
        # Specify the credentials of the repository, if required.
        # Use securityUtility command to encrypt the password.
        LARS.user=admin
        LARS.password=yourPasswordHere

Testing the connection to LARS

  1. Ensure that your Liberty runtime can talk to LARS: 
    cd ~/liberty-server/wlp/bin

./installUtility testConnection

    You’ll see messages like these confirming that the Liberty runtime is connected to LARS:

    Testing the connection to all configured repositories...
This process might take several minutes to complete.

Configured Repositories
----------------------------------------------------------------------
Name: LARS
Location: http://localhost:9080/ma/v1
Status: Successfully connected to the configured repository.

Name: IBM WebSphere Liberty Repository (Default Assets Repository)
Status: Connection to the repository is not tested.
Reason: The Default Assets Repository is not enabled in the following
    configuration file:
    /home/liberty/liberty-server/wlp/etc/repositories.properties

Install CDI 1.2 from LARS

After all this setup, you’re now ready to install some Java EE 7 features into your Liberty runtime. Remember, you downloaded the Liberty kernel runtime package, which is a minimal download that does not contain any runtime features. Let’s say you have an application that requires CDI 1.2. Your Liberty runtime currently can’t run that application. No problem: just install that feature from LARS:

    cd ~/liberty-server/wlp/bin

    ./installUtility install cdi-1.2

You’ll see messages like these confirming that the cdi-1.2 feature has been installed from LARS:

    Establishing a connection to the configured repositories...
    This process might take several minutes to complete.

    Successfully connected to all configured repositories.

    Resolving remote assets. This process might take several minutes to complete.
    ....
    ....
    ....
    One or more features installed successfully: cdi-1.2.
    Start product validation...
    Product validation completed successfully.

You can follow the same process to install any combination of Java EE 7 features from LARS.

Conclusion

In this post, we have gone through the process of setting up an on-premise repository and using it to install features into a Liberty runtime. This is just a simple example with LARS and one Liberty runtime on the same host but LARS can similarly be used by many Liberty runtimes. By running LARS, rather than the public Liberty Repository, you gain control over the precise set of application server features that are installed into your runtimes.
