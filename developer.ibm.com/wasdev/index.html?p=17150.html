Deploying a web application using Liberty, DB2, Docker Swarm and Docker Compose across multiple Docker machines

Multiple Docker machines linked through Docker Swarm running your web application is referred to as a multi-node Web application. Launching a multi-node web application using Docker Compose allows for easy single command deployment and scaling of web applications across multiple hosts. This article explains how to create a multi-node web application that uses a single DB2 database.

Spreading your containers across different machines in a multi-node web application can help you to achieve partial high availability (to achieve full high availability you will also need to follow these steps). Note that deploying across multiple machines using this configuration does not increase capacity unless you are using a load balancer such as IBM HTTP Server (IHS).

When you use Docker Compose to launch the multi-node web application, instances of the application are deployed to Liberty servers. The Liberty servers, alongside DB2, are each running in their own Docker container. Using Docker Swarm and Docker Compose’s networking functionality enables these containers to communicate across multiple hosts as well as the ability to manage the hosts as though they were a single entity. Using Docker Compose enables us to scale the web application easily using a single command to deploy additional instances of the same application.

Before you start

  • Register for a Docker Hub account.
  • Install two Linux machines or virtual machines with Docker, Docker Machine and Docker Compose.
  • Download the code for the sample.docker.db2 sample on GitHub.
  • Download the DB2 drivers and copy them to the app directory in the the sample.docker.db2 code you downloaded (the version of DB2 required can be found on the DB2 Docker Hub page).

Preparing the application

Prepare the sample.docker.db2 application with your Docker details and push the image to Docker Hub:

  1. Navigate to the app directory in the sample.docker.db2 code.
  2. Set your Docker user name:

    export DOCKER_USER=yourdockerusername

  3. Build the application:

    docker build -t ${DOCKER_USER}/app .

  4. Log in to Docker Hub:

    docker login --username=${DOCKER_USER}

  5. Push the image to Docker Hub:

    docker push ${DOCKER_USER}/app

Creating the swarm

  1. Navigate up a level:

    cd ../

  2. Run the createSwarm.sh script passing the number of agents required as the first argument. This creates a swarm using a consul key-store (find out more).

    ./createSwarm.sh 2

  3. Take control of the swarm-master:

    eval $(docker-machine env --swarm swarm-master)

Launching the multi-node web application

  1. Set the project name:

    export COMPOSE_PROJECT_NAME=yourprojectname

  2. Set your Docker Hub user name environment variables:

    export DOCKER_USER=yourdockerusername

  3. Create the application:

    docker-compose up -d

  4. Optionally scale the application up:

    docker-compose scale app=numberofinstances

  5. List the running containers:

    docker ps

  6. Navigate your browser to one of the running containers at <hostIP>:<containerPort>/TestDB/HelloWorld.

That’s it.

What configuration is happening under the covers?

DB2

The DB2 directory contains a single Dockerfile that uses the ibmcom/db2express-c image as a base, starts the db2 service, then creates a database called Test that the Web application will use to verify that it exists:

FROM ibmcom/db2express-c
RUN su - db2inst1 -c "db2start && db2 create db Test"

Application

The Dockerfile in the application directory manages all the other files in it. It uses the Java EE 7 Web profile websphere-liberty image as a base, then overwrites the server.xml using the equivalent file in the directory. It places the TestDB.war application in the dropins folder and then copies over both the DB2 drivers and their licenses into the active directory. The location of the drivers can be changed but the server.xml must be updated to reflect these changes:

FROM websphere-liberty:webProfile7
COPY server.xml /opt/ibm/wlp/usr/servers/defaultServer/
COPY TestDB.war /opt/ibm/wlp/usr/servers/defaultServer/dropins/
COPY db2jcc4.jar /
COPY jdbc_LI_en /

Docker Compose YAML

The first section of the YAML defines the services, first building the image from the db2 directory, accepting the license and setting the password before finally running the db2start command that starts up the service. The second service defines the application itself, pulling the previously pushed image from Docker Hub. It sets the DB2 port and host name as environment variables so that the server.xml file can pick them up then exposes the default ports. The project name and Docker user name environment variables are set by the export command. In the networks section of the YAML, the script simply changes the default network to use overlay drivers rather than bridge drivers:

version: '2'
services: 
  db:
    build: ./db2
    environment:
      LICENSE: accept
      DB2INST1_PASSWORD: thisisthepwd
    command: db2start
  app:
    image: ${DOCKER_USER}/app
    environment:
      DB2_PORT: 50000
      DB2_HOST: ${COMPOSE_PROJECT_NAME}_db_1
    ports:
      - "9080"
      - "9443"
networks:
  default:
    driver: overlay

That concludes this basic usage tutorial for launching web applications running on Liberty across multiple hosts using Docker, Docker Swarm, and Docker Compose.
