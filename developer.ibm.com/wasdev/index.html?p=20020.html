Use connection validation to check that pooled connections are still valid before using them, and connection initialization to run SQL commands when connections are created.

As part of our continuing effort to ease the migration of applications from WebSphere Application Server traditional (WAS) to WebSphere Liberty, weâre constantly adding new capabilities to Liberty that mirror existing functionality in WAS. For the Liberty November beta, we added two new functions to the existing JDBC features:

  • Connection validation
  • Connection initialization

Both are documented in the Liberty beta Knowledge Center but, to help you understand how to use this new functionality, Iâll cover them in a bit more detail and show you some config snippets.

Connection Validation

As of the Liberty November beta, you can now specify that the application server verifies that a pooled database connection is still valid before loaning it to your application. If you’ve developed applications for WAS, this is a configuration option that you might be familiar with already. What does valid mean in this context? It means that the connection to the database is still active and, if used, should not throw an exception complaining about the connection being closed, disconnected, or similar.

The benefits of connection validation

Why would you use connection validation? The application server pools JDBC connections to relational databases. This can provide significant application performance gains because creating a managed connection costs both resources and time. If your application created a new connection from scratch every time it needed one, it would be expensive, so the application server manages a pool of connections for you and will first opt to loan you a connection from the pool if it can instead of creating a new connection.

Like books from a library, connections are constantly being loaned from and returned to the pool. If a connection sits unused for too long in the pool, it might become ‘stale’ because the communication path that underlies each of the connections between the application server and the database has disconnected. The disconnection might be due to planned or unplanned networking problems, timeouts configured on the database server because of inactivity (e.g. overnight or during holidays when your application has no users), or a number of other causes. If the application attempts to use a stale connection, the communication will fail.

The frequency at which connections become stale (if ever) is determined by many factors that may be outside of your control. A well-behaved application must handle this situation, frequent or infrequent as it may be. You could add code to every place in your application that requests a connection in order to handle this failure and request another connection, but then youâve got to test and maintain that code forever. Or, you could rely on the app server to do that work for you by enabling connection validation. That sounds like a no-brainer, sign you up for connection validation, right?

Assessing the performance trade-offs of connection validation

Several things to consider first. Your applications must still handle stale connections properly because connection validation reduces, but doesnât eliminate, the possibility of your applications getting stale connections. You should also consider the overhead of performing connection validation. Liberty validates the connection using the standard approach required by the JDBC 4.0 spec (the Connection.isValid(int timeout) method) so the performance overhead is the amount of time it takes for a round-trip from the JDBC driver to the database over the network.

Since this overhead is incurred each time a pooled connection is returned to your application, youâll need to investigate whether it is significant enough to cause a performance problem. Depending on the database vendor and your network, the times will vary. In an unscientific test, I VPNâed from my home to our development lab on the other side of the USA. I used a simple app to call Connection.isValid() a number of times and, averaging the response time, I calculated that a DB2 JCC 4.19.49 JDBC driver took an average of 103.2 ms to complete validation, while an Oracle 12.1c JDBC driver took an average of 97.1 ms. Clearly, your private network will show substantially different round-trip timing but you can readily measure it as I did.

Liberty makes it very easy to get a sense of the amount of time that some of your JDBC operations are taking:

  1. Enable the Liberty event logging feature and configure by adding <feature>eventLogging-1.0</feature> to the features manager of your server.xml configuration file.
  2. Override the default value for minDuration to allow Liberty to log all SQL operations regardless of their length by adding <eventLogging minDuration=â0â></eventLogging> as a top-level XML element.

For each SQL operation that your application executes, youâll get a line detailing the length of time that the operation took, similar to this:

[INFO    ] END requestID=AAFNTvX2T6m_AAAAAAAAAAn # eventType=websphere.datasource.psExecuteQuery 
# contextInfo=jdbc/DictionaryDB | SELECT * FROM Dictionary WHERE WORD = ? # duration=100.006ms

This log record tells me that my app used a data source (with a JNDI name of jdbc/DictionaryDB) to execute a prepared statement query (psExecuteQuery) which took roughly 100 milliseconds. By default, event logging is enabled for all type of SQL operations (and HTTP requests); if youâre interested in timing information for only specific types of SQL operation, see the Liberty Knowledge Center for how to specify operation types.

When you know the length of time that your SQL operations take, you can decide whether reducing the possibility of applications getting a stale connection from the pool is worth any additional overhead of connection validation. Part of that choice could be influenced by the frequency with which you experience stale connections. If you encounter stale connections frequently because your network is flaky or because there is a short connection timeout period enforced by your database administrator, it might be worthwhile to let the application server handle it for you.

Enabling connection validation

So, assuming youâve decided to try out connection validation, itâs easy to do. Edit your server.xml file and add the validationTimeout attribute to any <dataSource> element using a JDBC 4.0-compliant driver like this:

<dataSource id="myDS" jndiName="jdbc/myDSâ validationTimeout="0">
...
</dataSource>

The timeout value that you specify determines how long the JDBC driver will wait for a response from the server. A value of 0 means youâre willing to wait forever. Like most configuration options in Liberty, once you save those changes to your server.xml file, connection validation will begin on the data sources for which you have enabled it.

Connection initialization

Connection validation is performed on pooled connections but connection initialization works on newly-created connections. As of the Liberty November beta, you can specify one or more SQL commands to be executed by the application server when creating a connection. Again, you might be familiar with using similar capabilities in WebSphere Application Server traditional.

When would you need to use connection initialization? It can be useful for a number of scenarios like, for example, setting the schema, time zone, or language so that your application doesnât have to. A common scenario is that you have an application that you want to be able to run using either a TEST or PRODUCTION schema without having to modify the app. By using connection initialization, you can easily configure the data source to set up the appropriate schema on each connection that it establishes.

For example, using Derby, you can issue the SQL command SET CURRENT SCHEMA = TEST to change the default schema for your connection to the TEST schema. Rather than having to issue that command somewhere in your application, configure the data source to issue it for you by adding the new child element onConnect to your data source element like this:

<dataSource id="myDS" jndiName="jdbc/myDSâ >
  <onConnect>set current schema = TEST</onConnect>
  â¦
</dataSource>

As another example, if you have an application that uses an Oracle database and you want to be able to control the formatting of dates across all SQL statements in that application, you could use connection initialization with multiple onConnect elements to specify the data format like this:

<dataSource id="myDS" jndiName="jdbc/myDSâ >
  <onConnect>alter session set NLS_DATE_FORMAT=âYY.MM.DDâ</onConnect>
  <onConnect>alter session set NLS_DATE_LANGUAGE=âCANADIANâ</onConnect>
  â¦
</dataSource>

So weâve discussed how to verify that pooled connections are still good using connection validation and how to customize new connections using connection initialization with examples of when you might use them. In addition to the IBM KnowledgeCenter documentation, hopefully this article provides you with a better understanding of the new JDBC functions added to the Liberty November beta.
