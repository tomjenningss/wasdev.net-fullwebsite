Passing the CTS tests on WAS Liberty


Kevin Sutter
Published on July 3, 2015 / Updated on September 25, 2015
0 Comments

CTS stands for Compatibility Test Suite. This is the bucket of tests that a Java EE Application Server is required to pass in order to state Java EE Compatibility and to be posted on the Java EE Compatibility web site.Â  We would not have been able to pass CTS in our abbreviated release cycle if it wasnât for the architecture and features in the WAS Liberty offering. A little background first

WAS Liberty has been a Java EE 6 Web Profile offering in the past. During the summer of 2014, we decided that we should graduate WAS Liberty to the Java EE Full Platform to be consistent with our WAS classic offering. And, we determined that it made no sense to go ‘backwards’ and just support Java EE 6 Full Platform. So, letâs move forward and support Java EE 7 Full Platform. And, letâs also do this on an abbreviated delivery schedule of 2Q2015â¦

As many of you realize, we delivered some of the Java EE 7 features early via our Continuous Delivery mechanism. We delivered a handful of Java EE 7 features in 4Q2014 and another handful in 1Q2015. The requirement for delivering these individual features is to pass the featureâs TCK (Technology Compatibility Kit). These TCKs are normally ‘standalone’ test efforts with, hopefully, minimal dependencies on other Java EE 7 features.

The CTS bucket combines all of these individual TCKs along with additional integration tests to ensure a cohesive Java EE offering. Dependencies across Java EE technologies are not only allowed, but expected. The full CTS bucket is much more comprehensive than the individual TCKs. It contains over 75,000 tests — all of which need to pass to claim Java EE Compatibility!

Small, nimble architecture

Each of the CTS tests required applications to be deployed, started, executed, and verified. In some cases, the applications are also undeployed during the testing process. And, although not a CTS requirement, cycling of the application server was required during the debugging phases of various testcase failures.

The CTS team was also able to take advantage of the portability of the WAS Liberty server configuration. They were able to quickly install new WAS Liberty test images without having to manually reconfigure the server, or write and maintain complicated configuration scripts. They simply copied a saved server directory into a new installation, and the tests were up and running! Additionally, testers were able to quickly share updated server configurations across machines with little or no updates to the server.xml.

You can easily see how the execution of these 75,000 tests could take an enormous amount of time and resource with a normal application server. But, with WAS Libertyâs super fast server startup, simple application deployment and configuration, and our herculean CTS test team, we made tremendous progress on this CTS bucket in a very short amount of time.

Debugging made simple

On the rare occasion where we actually hit a bug when executing the CTS tests (smile), another benefit of the architecture of WAS Liberty became apparent. It was very easy to isolate the testcase failure and to share the failure with the affected teams. Whether this was collecting the logs, or collecting the server and application configuration, or even to share the execution environment â the easy packaging of WAS Liberty during these debug cycles helped us stick to our schedules.

We use the same basic set of tools for developing WAS Liberty as we recommend to our customers that use WAS Liberty â Eclipse with WDT (WebSphere Development Tools). This simple setup also makes it easy for our test teams and development teams to share information when debugging issues. Whether an individual is using Windows, or Linux, or a Mac â it doesnât matter. Sharing of development, test, and debug information is the same basic process.

Plug for RTC (Jazz)

Another example of ‘eating our own dog food’ is that we use RTC (Rational Team Concert) to manage our WAS Liberty source code. The use of RTC allowed us to easily share change sets across components so that the various teams were all running with the same set of test patches. As we were developing code fixes for the various CTS tests, it was very easy to know which patches helped and which ones were not so good… And, we could determine this before delivering the code changes to the greater development community.

Summary

Just a few short weeks before our CTS deadline, some of our experienced CTS testers figured we would never make it. We had several hundred test failures with very little time left in the schedule. As we started to make dent after dent in this test failure backlog, and the CTS light at the end of the tunnel was getting brighter, I started to think about how we could be making these huge strides — especially when our veterans were saying it couldnât be done. The only answer that made sense was the use of WAS Liberty!
