On Friday 15th June 2012, WebSphere Application Server V8.5 was released. This release includes many new capabilities and bug fixes over the beta. The latest version of the Liberty profile can be downloaded for no charge from our download page. This page contains the new and noteworthy features of the Liberty profile and WebSphere Application Server Developer Tools for Eclipse since the March Refresh of the beta.

Java Naming and Directory Interface

  • The jndi-1.0 feature enables the configuration in server.xml of simple properties to be made available in the JNDI namespace.
  • Data sources are now available directly through the JNDI interface without requiring a resource reference.
  • You can now query the java:comp namespace using the listBindings method of javax.naming.Context.

REST

  • The jax-rs-1.0 feature enables the use of JAX-RS 1.0 from within applications.
  • The json-1.0 feature enables the use of the JSON4J interfaces for dealing with JSON formatted messages.

JDBC

  • Data sources can now be defined in Java EE applications either through the deployment descriptor, or through annotations.

Monitoring

  • The monitor-1.0 feature enables performance statistics to be gathered.
  • Performance statistics can be gathered from MX MBeans available in the MBean service.

OSGi applications

  • JPA is supported for Web Application Bundles.
  • JSF is supported in Web Application Bundles.
  • TagLibs can now be used from Web Application Bundles.

Production

  • The Liberty profile is now supported for use in production.
  • Sessions can be persisted to a database using the sessionDatabase-1.0 feature. This allows session failover between servers.
  • Sessions can be persisted using WebSphere eXtreme Scale.

Build

  • Ant tasks are provided that can be used to start and stop the Liberty profile and to deploy and undeploy applications.
  • A maven repository is available that provides the API jars for the Liberty profile and plugins for starting and stopping liberty servers as well as allowing applications to be deployed and undeployed.
  • The dev directory contains all the Java APIs required to compile applications to be deployed to the Liberty profile. Jar files are stored in one of three directories: spec, ibm-api, third-party. The spec directory contains APIs provided by standards bodies like the JCP or the OSGi Alliance. The ibm-api directory contains APIs that provide additional capabilities beyond standards-based APIs. The third-party directory contains non-standard programming interfaces provided by organizations other than IBM, such as open source programming interfaces.

Classloading

  • The Liberty profile classloaders for Java EE applications hide the classes that constitute the internal implementation of the Liberty profile. This provides several benefits including:
    • Applications cannot accidentally use internal unsupported programming interfaces.
    • Open source libraries that make up the server runtime do not bleed through to applications. For example the Liberty profile includes the SLF4J logging library at version 1.5.6. An application that makes use of SLF4J version 1.6.6 can do so without interference with the runtime’s copy.
    • Classes can be found faster because the product internals are not searched when loading application classes.
  • The Liberty profile classloader can be configured to show and hide various categories of classes. The categories supported are: spec, ibm-api, third-party. By default spec and ibm-api packages are visible, however this can be expanded or shrunk.
  • Applications can now share class instances from libraries between applications. This is achieved using a commonLibraryRef.

Security

  • Security role mappings in the server.xml file are merged with the ibm-application-bnd.xml file. The copy in the server.xml file takes precedence on a per role basis.
  • Simplified ssl configuration.
  • Support for the Java EE security annotations within Web applications.
  • The user password for run-as role bindings is not required.

Beta Restrictions

The following restrictions from the beta have been removed from the Liberty profile:

  • The Liberty profile can be run using Java 7.
  • Filesets now support comma-separated lists of includes.
  • The install directory name and path can contain spaces.
  • Logging properties are correctly processed.
  • FFDC log files are filtered and cleaned up automatically.
  • The web server plug-in configuration sets the web server ports automatically.

All restrictions associated with the following features have been removed:

  • blueprint-1.0
  • jdbc-1.0
  • servlet-3.0
  • wab-1.0
  • zosTransaction-1.0

Changes from the Beta

The following are major changes that have occurred since the beta that will affect developers migrating from the beta.

  • The Liberty profile no longer ships with a default server.
  • When creating a new server the server is configured so it can run Web applications using JSP 2.2.
  • The syntax has changed for the securityUtility createSSLCertificate command. For the beta, parameter names included a leading single hyphen. Now they include a double hyphen. For example:
    (beta):
    securityUtility createSSLCertificate -server myserver -password mypassword -validity 365 -subject CN=mycompany,O=myOrg,C=myCountry
    (now):
    securityUtility createSSLCertificate --server myserver --password mypassword --validity 365 --subject CN=mycompany,O=myOrg,C=myCountry
  • The webSecurity-1.0 and security-1.0 have been removed and replaced with the appSecurity-1.0 feature.
  • The jmx-1.0 feature has been removed and replaced with the localConnector-1.0 and restConnector-1.0 features.
  • The trace specification attribute in the server configuration file has been renamed from trace.specification to traceSpecification.
  • To configure the use of a shared library, you use the classloader element. The libraryRef attribute of the classloader element has been renamed as commonLibraryRef.
  • The property for disabling monitoring of the dropins directory is renamed from locationMonitoringEnabled to dropinsEnabled.
  • For defining the properties of the Derby embedded datasource, you can now use the properties.derby.embedded element.
  • For enabling SSL client authentication, the keystores element is renamed to keyStore, and the repertoire element Â is renamed to ssl.
  • SSL configuration is significantly simplified from the beta. As a result the defaultSSLConfig.xml is no longer necessary to get SSL working.
  • For the beta, if the authorization configuration for the EAR file is defined in an ibm-application-bnd.xml file and also in the server.xml file, then the version in the server.xml file is used. Now, the two configurations are merged. If there are conflicts, the configuration specified in the server.xml file is used.
  • The following attributes of the trustAssociation element are renamed:
    • performTAIForUnprotectedURI is renamed to invokeForUnprotectedURI
    • allowAppAuthMethodFallback is renamed to failOverToAppAuthType
    • taiProperties is renamed to properties
  • The monitorConfiguration attribute of the config element is replaced with the updateTrigger attribute.
  • Server dumps and server packages are generated into the server output directory by default.
  • Server dumps are generated using a unique file name, so multiple calls to the dump tool will result in new dump archive files rather than replacing the previous one.
  • The location of the DB2 libraries on z/OS is specified using a library element in the server configuration rather than using the STDENV file in the server’s JCL procedure.
