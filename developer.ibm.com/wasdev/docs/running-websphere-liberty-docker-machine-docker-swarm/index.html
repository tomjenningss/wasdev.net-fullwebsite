Docker is useful if you’re developing applications to run on Liberty or if you’re working in DevOps to deploy those Liberty applications into test or production environments. The web application is deployed to a Liberty server in a Docker container. That container can then be replicated and deployed into any or all of on-premise, cloud, and hybrid cloud environments. Docker Swarm makes it easier to run a number of hosts and manage them as a single entity. Docker Machine provides an easy way to create those hosts.

Update 28th July 2016: If you are using Docker 1.12 or later, please see the following updated tutorial: Deploying Liberty applications using Docker 1.12 and swarm mode

In this tutorial you will learn how to develop a simple Docker image containing WebSphere Liberty and then run it across multiple hosts. You will use Docker Machine to quickly and easily stand up a Docker Swarm cluster that can be managed by a single endpoint. You will then use a simple script to deploy multiple WebSphere Liberty-based applications such that those application instances are distributed across the swarm in a way to maximize availability of the applications to your users in the event of failures.

This demonstrates how, in developing the initial image, you do not need to take in to account how it will be subsequently deployed. The image could be deployed on your local development machine as we do here, in a test environment, or deployed out into production in the cloud, on-premise, or in a hybrid cloud scenario.

Docker Swarm and Docker Machine

Docker Swarm is a tool that enables you to run a cluster of Docker nodes and manage them as a single entity, a swarm. Docker Machine provides an easy way to create those nodes with Docker installed and configured to run as a swarm.

In this tutorial we’ll be using Docker Machine to provision VirtualBox images on your local machine. Using VirtualBox means you can run Docker in a virtual machine on systems that do not have a Linux kernel, e.g. your Windows or OS/X machine. You can also use Docker Machine to create virtual machines on many cloud platforms, including IBM SoftLayer.

When deploying applications to a swarm there are a number of strategies provided out-of-the-box for distributing application instances across the nodes. The spread strategy attempts to distribute workload evenly across the nodes and the binpack strategy attempts to minimize fragmentation. Neither of these strategies, nor the random strategy, attempts to ensure that instances of any given application are distributed across the swarm in order to maximize availability in event of failure of a node. With this tutorial we have provided a script that implements this behavior.

Before you start

  • Install Docker and Docker Machine. On Windows and Mac this is most easily achieved by installing the Docker Toolbox.
  • Create a Docker Hub account.

Creating and launching the swarm

Using the Docker command-line, if it doesn’t already exist create a VirtualBox virtual machine on your system called default:

docker-machine create -d virtualbox default

Creating the swarm with a script

You can optionally automate the remaining steps in this section by running the following script, swarm.sh, passing the number of agent nodes required as a parameter (e.g. bash swarm.sh 3):

#! /bin/sh
if [ $# -eq 0 ] then
  echo "No arguments... Pass the number of agents required"
  exit 1
fi
token=$(docker $(docker-machine config default) run --rm swarm create)
echo "Swarm token is $token."
echo "Creating Swarm Master"
docker-machine create -d virtualbox --swarm --swarm-master --swarm-discovery token://$token swarm-master
for (( i=1; i<=$1; i++ )) do
  echo "Creating Swarm Agent $i" docker-machine create -d virtualbox --swarm --swarm-discovery token://$token swarm-agent$i
done
echo "Node IPs: " docker $(docker-machine config default) run --rm swarm list token://$token
echo "To control this Swarm run: 'eval $(docker-machine env --swarm swarm-master)'"

Creating the swarm manually

To manually create the swarm, use Docker Machine to configure your environment so that your local Docker client will communicate with the Docker daemon on the machine ‘default’.

eval $(docker-machine env default)

Generate a swarm discovery token from Docker Hub by running the swarm create command in a container. This token should then be used when creating the swarm master and agents. (You could, alternatively, use a distributed store such as Apache ZooKeeper, etcd or Consul as the discovery service.)

docker run swarm create
Output: b806ae3cf6b09f04c60e0070a2c85bd3

Using Docker Machine, create a swarm manager with the VirtualBox driver and the swarm discovery token that you created earlier:

docker-machine create -d virtualbox --swarm --swarm-master --swarm-discovery token://b806ae3cf6b09f04c60e0070a2c85bd3 swarm-master

Finally, create two (or as many as you require) agent nodes, again using the swarm discovery token:

docker-machine create -d virtualbox --swarm --swarm-discovery token://b806ae3cf6b09f04c60e0070a2c85bd3 swarm-agent1
docker-machine create -d virtualbox --swarm --swarm-discovery token://b806ae3cf6b09f04c60e0070a2c85bd3 swarm-agent2

Application images

For this tutorial we will just tag two identical WebSphere Liberty Docker images to act as our different application but you may use your own application images instead.

  1. Log in to your Docker Hub account:

    docker login

  2. Using the Dockerfile provided below, ensure you are in the correct directory and then run the following commands to create two Liberty images with different names where <NAMESPACE> is your namespace (account ID) on Docker Hub:

    docker build -t <NAMESPACE>/app1 .
    docker build -t <NAMESPACE>/app2 .

    Use this Dockerfile:

    FROM websphere-liberty
ENV LICENSE accept

    The Dockerfile simply builds a new image from the existing websphere-liberty image using a different name each time to imitate two different applications and accepts the license.

  3. Push the two images to Docker Hub using the following commands:
    docker push <NAMESPACE>/app1
    docker push <NAMESPACE>/app2

Running your Liberty application in a container on the swarm

The provided script addApp.sh distributes applications on the swarm by checking what images are running on each node and picking the node which contains the least number of occurrences of an image. This ensures that, unlike any of the out-of-the-box strategies, for any given application it is distributed across multiple nodes.

To take control of the swarm, point to the swarm master by loading the swarm-master machine configuration, this time adding the --swarm flag:

eval $(docker-machine env --swarm swarm-master)

To add your application, simply run the addApp.sh script:

./addApp.sh <NAMESPACE>/app1
./addApp.sh <NAMESPACE>/app2

The addApp.sh script contains the following code:

#! /bin/sh
if [ $# -ne 1 ]
  then
  echo "Usage: addApp.sh [IMAGENAME]"
exit 1
fi
nodes=$(docker-machine ls --filter swarm=swarm-master -q)
for node in $nodes do
  instances=$(docker $(docker-machine config $node) ps --filter ancestor=$1 -q | wc -l)
  containers=$(docker $(docker-machine config $node) ps -q | wc -l)
  if [ -z "$minNode" ] || [ $instances -lt $minInstances ] || ( [ $instances -eq $minInstances ] &amp;&amp; [ $containers -lt $minContainers ] ); then
    minNode=$node
    minInstances=$instances
    minContainers=$containers
  fi
done
echo "Running $1 on node $minNode"
docker run -d -P -e constraint:node==$minNode $1

The script requires one parameter of the name of the image that contains your application (line 4).

The script uses the image name to check for any other running instances of that image on the swarm and finds the node with the least number of running occurrences. If all nodes have the same number of a particular image then the script will place the application on the node with the least number of containers.

Removing the swarm

When you have finished this tutorial, you can remove all of the machines in the swarm using the following command:

docker-machine rm $(docker-machine ls --filter swarm=swarm-master -q)

Congratulations – you have created a simple Docker image containing WebSphere Liberty and deployed it across multiple Docker hosts! You have also seen how a short script can be used to ensure that instances of each application are balanced across the multiple hosts in a way to ensure maximum availability of your application.
