Developers often have limited hardware resources at their disposal. Efficient usage of these resources becomes an absolute requirement. Sharing resources between multiple applications is one of the ways to achieve this and multitenancy, with varying degrees of isolation, can be achieved at a number of different levels in either hardware or software. This article describes how the MT-JVM can be transparently exploited by web applications deployed to the lightweight WebSphere Application Server Liberty Profile.

For Java applications, the Java Virtual Machine (JVM) in which the application runs, can be shared between multiple isolated applications. The applications are â€˜tenantsâ€™ of the shared JVM to reduce overall memory consumption. The advantages of using a multitenant JVM are described in further detail at Introduction to Java multitenancy. You can try out the multitenant feature of the IBM SDK (MT-JVM) in a technology preview in IBM SDK version v7R1 (for non-production scenarios).

With Java multitenancy, you can optionally share the JVM among applications that traditionally were using separate JVMs. This gives the benefit of reduced memory consumption and processing costs, while providing static field isolation for individual tenants as well as per-tenant control over processor time, heap size, thread count, file and socket I/O (read/write bandwidth). MT-JVM can be used in conjunction with other levels of multitenancy that virtualize the hardware and operating system (such as Linux containers).

Scenarios that can benefit from running Liberty in MT-JVM are described below.

Components of MT-JVM

To run an application as a tenant that shares a JVM, the -Xmt option needs to be passed to the java launch command. The presence of the â€“Xmt option instructs the Java runtime to inject the user application into a JVM daemon (javad) rather than creating a new JVM. The heap is shared across tenants and resides in the javad address space. Using the balanced garbage collection, the heap is split into regions and each region is associated with a tenant. Each tenant will only use the regions that have been associated with it. The result is that each tenant can have a set of regions (memory) dedicated to them; however, the actual collection of garbage objects is done for all tenants when garbage collection is started. The daemon process is configured using a javad.options file.


Running WebSphere Liberty in MT-JVM

Any version of the WebSphere Liberty server can be launched as a tenant in the MT-JVM; the MT-JVM is available for download. To evaluate MT-JVM with Liberty, for each Liberty server that will be a tenant, you must specify a minimal amount of configuration in its jvm.options file to direct the Java runtime to launch the server as a tenant (see the documentation for the jvm.options file).

Specify the following properties to run a server as a tenant:

  • -Xmt – Runs the server as a tenant in a multitenant JVM.
  • -Djavad.home=<absolute path to the folder in which javad.options file resides> – Specifies which multitenant JVM to run in. All Liberty servers that need to share the same MT-JVM must point to the same folder path that contains the javad.options file.

For example:

-Xmt
-Djavad.home=/home/libertyadmin/javad

Each Liberty server is started as a tenant in the Java MT architecture. All liberty servers that intend to share the same daemon (JVM) should point to the same folder where the javad.options file resides.

The javad.options file must have the following entries at a minimum:

  • The following three properties are mandatory for a Liberty Server to be started in the multitenant mode using the /bin/server command:
    • -Xtenant
    • -Xrcm:consumer=autotenant
    • -Dcom.ibm.cloud.javaagent=yes
  • -Djavad.persistAfterEmptyTime=0 – Causes the JVM server daemon to end as soon as the last tenant disconnects/shuts down (recommended).

For example:

-Xtenant
-Xrcm:consumer=autotenant
-Xmx=256m 
-Djavad.persistAfterEmptyTime=0
-Dcom.ibm.cloud.javaagent=yes

Some examples where MT-JVM provides benefit for WebSphere Liberty

Scenario 1 (optimizing server start and stop times)

Consider a scenario in which an automated test suite needs to start and stop several Liberty servers frequently. Using MT-JVM reduces start-up times over a period of time as the daemon (JVM) is already created and the server just needs to connect to that daemon. This eliminates the overhead of creating a new JRE.

Scenario 2

Consider a scenario in which web applications need to be tested with various Java locales. When two applications are running in the same JRE, changing the locale in one application can affect the output of another application if it relies on a different locale. MT-JVM provides the required isolation so that a change to the locale in one application does not affect the locale in another application, even though they share the same JRE. This scenario highlights the static isolation that is provided by MT-JVM.

Scenario 3

Consider a scenario where a large number of small web applications are suitable to run in the same address space but need more isolation from each other than can be provided in a single Java EE application server. Several applications can be deployed in the same JVM with MT-JVM.

Limitations

MT-JVM has certain limitations, including lack of support for native libraries accessed by tenants across JNI. For more details see the official documentation.
Note that IBM Java MT is provided only for evaluation purposes and not for use in a production environment.

Summary

WebSphere Liberty servers can run as isolated tenants inside a shared MT-JVM. This reduces overall resource consumption across multiple isolated web applications. In lab tests comparing 16 tenants in MT-JVM to 16 standalone JVMs running the same workload, both using shared class cache and the balanced GC policy, MT-JVM provided an overall footprint reduction of 13% with a 17% faster start-up time (although slower for the first tenant).

The footprint and startup times of WebSphere Liberty are already small, so this is a 17% improvement on only 3.5 seconds start-up time for the test workload. There is a throughput performance cost for MT-JVM which, for the benchmark workload, resulted in 9% better throughput using standalone JVMs.

MT-JVM works with any version of WebSphere Liberty. For evaluation purposes weâ€™d recommend using the latest stable release of Liberty.
